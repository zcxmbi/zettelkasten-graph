# 游戏开发→数学 笔记模板

> 适用场景：游戏开发中遇到一个具体问题，分析其背后的数学原理和多种解法。

## 笔记结构

```markdown
---
tags: [游戏开发, 数学, <具体数学分支>]
---

# <问题标题>

> 一句话描述：游戏需要什么效果，当前卡在哪里。

## 一、具体问题

- 游戏上下文：属于哪个系统/机制
- 输入：什么数据进入
- 输出：期望产出什么
- 约束：性能限制、确定性要求、边界条件
- 当前实现（如有）：哪里不满意

## 二、数学分解

将问题拆为独立的数学子问题，每个子问题有清晰的 I/O：

| 子问题 | 数学本质 | 输入 | 输出 |
|--------|---------|------|------|
| A | ... | ... | ... |
| B | ... | ... | ... |

## 三、数学路线对比

对每个子问题，列出至少 2 种数学路线：

| 路线 | 核心方法 | 优点 | 缺点 | 性能 |
|------|---------|------|------|------|
| A.1 | ... | ... | ... | O(...) |
| A.2 | ... | ... | ... | O(...) |

## 四、选择方案

- 选哪条路线，为什么
- 关键公式/伪代码
- 参数说明和调参指南
- 性能优化（如降维、预计算）

## 五、相关概念

- [[数学概念A]] — 简述关联
- [[数学概念B]]
- [[相关设计文档]]

## 六、待探索

- 延伸问题
- 边缘情况未覆盖的
```

## 关键原则

1. **先拆后解**：别试图一个解法覆盖所有——先分解成独立的数学子问题
2. **路线对比 > 单一路线**：列出多种解法再选，笔记价值在对比过程
3. **性能显式标注**：游戏开发中算法复杂度直接决定可用性
4. **链接到数学概念笔记**：每个涉及的数学概念（正态分布、Simplex 噪声、Voronoi 等）各写一条原子笔记，再用 `[[wiki链接]]` 串联

## 避免

- 不把问题描述和数学混在一起写——先讲清楚问题，再进入数学
- 不省略"为什么选这个方案"——这是笔记的核心价值
- 不写成代码手册——数学笔记讲原理，具体实现放项目代码
